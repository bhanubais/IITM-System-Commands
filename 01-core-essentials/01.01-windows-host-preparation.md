# 01.01 Windows Host Preparation

## Purpose

Prepare the Windows host before Linux work starts.

Install and verify the package manager and PowerShell runtime used to manage WSL.

## Strict Syntax

```powershell
winget search <package>
winget show <package_id>
winget install <package_id>
winget upgrade <package_id>
winget uninstall <package_id>
pwsh --version
```

## Command Map

| Command | Use |
|---|---|
| `winget search` | Find package candidates. |
| `winget show` | Inspect package metadata before install. |
| `winget install` | Install a package. |
| `winget upgrade` | Upgrade one package. |
| `winget list` | Audit installed packages. |
| `winget uninstall` | Remove a package. |
| `pwsh --version` | Verify PowerShell availability. |

## Atomic Examples

```powershell
winget search PowerShell
winget show Microsoft.PowerShell
winget install Microsoft.PowerShell
winget upgrade Microsoft.PowerShell
winget uninstall Microsoft.PowerShell
winget list --source winget
winget --version
pwsh --version
winget upgrade Microsoft.AppInstaller
```

## Compound Scenarios

Audit package identity before install:

```powershell
winget search PowerShell
winget show Microsoft.PowerShell
winget install Microsoft.PowerShell
```

Refresh host package tooling:

```powershell
winget --version
winget list winget
winget upgrade Microsoft.AppInstaller
```

Validate shell readiness for WSL administration:

```powershell
pwsh --version
winget list --source winget
winget upgrade Microsoft.PowerShell
```

## Exam Traps

`winget search` finds candidates. It does not install.

`winget show` is the inspection step.

Package IDs are safer than display names when commands must be repeatable.

## Production Notes

Inspect package metadata before install.

Keep `Microsoft.AppInstaller` current because it provides `winget`.

Verify `pwsh` before WSL automation.
