# 08.01 Network Diagnostics

## Purpose

Inspect interfaces, reachability, routes, DNS, domain registration, and listening sockets.

## Strict Syntax

```bash
ip addr show
ifconfig
ping <host>
traceroute <host>
netstat -tulpn
nslookup <domain>
whois <domain>
```

## Command Map

| Command | Use |
|---|---|
| `ip addr show` | Show network interfaces and addresses. |
| `ifconfig` | Legacy interface inspection. |
| `ping` | ICMP reachability check. |
| `traceroute` | Show network path hops. |
| `netstat -tulpn` | Show listening TCP/UDP sockets and owning processes. |
| `nslookup` | Query DNS records. |
| `whois` | Query domain registration data. |
| `less` | Page long diagnostic output. |

## Atomic Examples

```bash
ip addr show
ifconfig
ping example.com                      # ICMP reachability
traceroute example.com                # path hop sequence
netstat -tulpn                        # TCP, UDP, listening, program, numeric
nslookup example.com                  # DNS query
whois example.com                     # registration data
whois example.com | less              # page long output
```

## Compound Scenarios

Check basic connectivity:

```bash
ip addr show
ping example.com
traceroute example.com
```

Inspect listening services:

```bash
sudo netstat -tulpn
```

Check name resolution and registration:

```bash
nslookup example.com
whois example.com | less
```

## Exam Traps

`ping` confirms ICMP response, not application correctness.

DNS success does not prove port reachability.

`ifconfig` may be absent on modern systems unless `net-tools` is installed.

`netstat -tulpn` may require privilege to show process names.

## Production Notes

Start with local interface state, then DNS, then route, then listening sockets.

Use pagers for long WHOIS output.

Correlate sockets with process IDs before restarting services.
