# 09.01 Git Fundamentals

## Purpose

Track source history, isolate work, and recover changed files.

Git protects working code when commits are made before risky changes.

## Strict Syntax

```bash
git init
git clone <url>
git status
git add <file>
git commit -m "message"
git branch <name>
git checkout <name>
git remote add origin <url>
git push -u origin <branch>
git pull
```

## Command Map

| Command | Use |
|---|---|
| `git init` | Create repository in current directory. |
| `git clone` | Copy remote repository locally. |
| `git status` | Show working tree and staging state. |
| `git add` | Stage file content. |
| `git commit` | Record staged content. |
| `git branch` | List or create branches. |
| `git checkout` | Switch branches or restore paths. |
| `git remote add` | Register remote repository. |
| `git push -u` | Push branch and set upstream. |
| `git pull` | Fetch and integrate upstream changes. |

## Atomic Examples

```bash
git init
git clone https://example.com/repo.git
git status
git add README.md
git add .                             # stage all changes under current directory
git commit -m "Add notes"
git branch feature-docs
git checkout feature-docs
git checkout main
git remote add origin https://example.com/repo.git
git push -u origin main               # set upstream for future pushes
git pull
git checkout -- broken_script.py      # discard uncommitted file changes
```

## Compound Scenarios

Create a local repository and commit:

```bash
git init
git status
git add README.md
git commit -m "Initial commit"
```

Create and publish a branch:

```bash
git branch feature-docs
git checkout feature-docs
git push -u origin feature-docs
```

Restore a modified file from the last commit:

```bash
git status
git checkout -- broken_script.py
git status
```

## Exam Traps

`git add` stages content. It does not commit.

`git commit` records staged content only.

Branches are pointers to commits.

`git checkout -- file` discards uncommitted changes to that file.

## Production Notes

Run `git status` before repair commands.

Commit known-good states before risky edits.

Use branches to isolate work from mainline code.
