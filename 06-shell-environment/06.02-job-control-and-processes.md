# 06.02 Job Control and Processes

Execute strict process scheduling and suspension operations. Shift executing commands seamlessly between foreground and background run states.

## Real World Scenario: Remote SSH Persistence
An administrator initiates a server backup using `tar` over an SSH connection. Halfway through the 40GB compression, power flickers at their local workstation. Because the process was tied to the shell, the SSH connection drops and the backup terminates abruptly, corrupting the archive. To prevent this, the administrator now starts the backup, suspends it with `Ctrl`+`Z`, pushes it to the background using `bg`, and completely detaches it from the shell tree using `disown`. Even if SSH drops, the server kernel natively finishes the compression payload.

## Job Governance

* `<command> &` : Deploy a process instantly in the background. Terminal control returns immediately.
* `coproc <command>` : Deploy a co-process utility sequence structure linked dynamically against the active shell.
* `Ctrl`+`z` : Suspend the active foreground operation and pause execution.
* `jobs` : Audit all suspended and running background jobs.
* `bg <%id>` : Transition a suspended job specifically to running status in the background.
* `fg <%id>` : Pull a background or suspended job strictly into the active foreground.
* `disown <%id>` : Decouple a job from the shell process tree. The process will persist structurally after shell termination.

### Detaching jobs safely - `disown -h`
Use `disown -h` to mark a job so the shell will not send it a SIGHUP when the shell exits. This is useful when you want a background job to continue after you log out or your SSH session drops, but you prefer not to remove the job from the shell job table.

Behavior summary:
- `disown` without flags removes the job from the shell job table and prevents the shell from sending SIGHUP when it exits.
- `disown -h` keeps the job in the job table but prevents SIGHUP from being sent to that job when the shell exits.

Example - detach an already-running background job safely:
```bash
# Start a long-running command, redirecting its output
long_backup_command > /var/log/backup.log 2>&1 &

# Identify the job id (for example %1)
jobs -l

# Prevent SIGHUP being sent if the shell exits
disown -h %1

# Now it remains listed in jobs but will not receive SIGHUP on shell exit
jobs
```

Important notes:
- `disown -h` only affects whether the shell sends SIGHUP. It does not redirect or close the job's file descriptors. For full detachment also redirect stdin/stdout/stderr (or use `nohup` or `setsid`).
- `disown` is a shell builtin in Bash and Zsh. It is not POSIX. For scripts that must be portable use `nohup` or start the job under `setsid` or `daemon` utilities.
- To detach all jobs at once use `disown -a` (shells that support the flag). Test behavior in your target shell before relying on it in production.


## Process Auditing

* `top -d 1` : Stream a live 1-second interval execution summary of all active processes. Press `q` to abort.
* `kill <id>` : Terminate execution nodes linked specifically securely directly to the current runtime identity profile.
* `ps auxw` : Output a static comprehensive list of all running processes across all users.
* `ps w --sort size` : Output operations sorted cleanly by memory consumption sizes.

### `ps` command common flags

| Flag | Description |
| :--- | :--- |
| **`-e`** / **`-A`** | Displays all processes on the system. |
| **`-f`** | Displays full-format listing (includes UID, PPID, and start time). |
| **`a`** | Lists processes for all users associated with a terminal. |
| **`u`** | Displays a user-oriented format, showing CPU and memory usage. |
| **`x`** | Includes processes without a controlling terminal (e.g., background daemons). |
| **`-u <user>`** | Filters processes to show only those owned by a specific user. |
| **`-p <PID>`** | Displays information for a specific process ID. |
| **`-o <cols>`** | Customizes the output columns (e.g., `-o pid,comm,%mem`). |
| **`--forest`** / **`f`** | Displays processes in a hierarchical tree format to show parent-child links. |
| **`ww`** | Wide output format that prevents long command strings from being truncated. |

### Real world use cases

* `ps aux | grep 'app_name'` : Search for a specific running application to find its process ID (PID) for troubleshooting or termination.
* `ps aux --sort=-%cpu | head -n 6` : List the top 5 highest CPU-consuming processes currently draining system processing power.
* `ps aux --sort=-%mem | head -n 6` : List the top 5 highest memory-consuming processes to quickly identify active RAM hogs.
* `ps -ef | grep nginx` : Verify if a specific background service or daemon (like Nginx) is actively running on the system.
* `ps -u username` : Audit a specific user account by listing every active process running under their permissions.
* `ps auxf` : Display a visual tree structure of all active processes to easily trace parent-child relationships.
* `ps -x` : Isolate and display background processes that are running silently without an attached terminal window.
* `ps -eo pid,%cpu,%mem,cmd` : Output a streamlined, custom table showing only the specific performance metrics you choose to monitor.
* `ps -p 1234` : Inspect the status, user ownership, and command details of one specific process ID.
* `ps auxww` : Force the terminal to wrap text instead of truncating long launch commands and file paths.

## Emergency Kernel Recovery

If a program fails to release the execution buffer and keyboard input ceases to appear, perform a terminal state reset.
1. Strike `Ctrl`+`J`.
2. Input `reset`. The keystrokes will remain invisible on the hardware layer.
3. Strike `Ctrl`+`J` to execute the reset binary and restore control.
