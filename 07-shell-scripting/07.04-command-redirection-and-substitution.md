# 07.04 Command Redirection and Substitution

Engineer precise control over input and output execution channels. Route payload data strictly via POSIX standardized system components.

## Real World Scenario: Silent Crontab Failures
A developer configured an automated `rsync` backup inside `cron` using `rsync -a /var/ /backup/ > /var/log/backup.log`. A month later, the database failed completely. When the developer checked `backup.log`, it was completely empty. The redirection (`>`) only captured `stdout`. The `rsync` operation failed dynamically because of permission errors, but the error bounds (`stderr`) vanished securely. To fix this, they re-engineered the shell execution strictly mapping both descriptors into the output queue: `rsync -a /var/ /backup/ &> /var/log/backup.log`.

## Logical File Descriptors
* `0` : Assigned to Standard Input (`stdin`).
* `1` : Maps precisely to Standard Output (`stdout`).
* `2` : Regulates explicit error tracing Standard Error (`stderr`).

## Chained Command Sequencing
* `cmd1 ; cmd2` : Execute instructions consecutively independent of failure paths.
* `cmd1 && cmd2` : Execute `cmd2` safely only if `cmd1` operates successfully.
* `cmd1 || cmd2` : Trigger `cmd2` exclusively as an error mitigation reaction if `cmd1` suffers structural failure.

## Output Stream Relocation
Separate clean operation outputs from error failures to maintain secure logging validation pipelines.

* `>` : Overwrite the mapped target node totally capturing the active datastream. Example: `script.sh > output.txt 2> error.log` separates logic failures securely.
* `>>` : Append payload cleanly ignoring existing file blocks and retaining data states.
* `2> fail.log` : Split operation stderr explicitly writing straight into localized debug tables.
* `&>` : Redirect the active stdout structure combined simultaneously with the stderr log structure.

## Linear Command Substitution

Command substitution captures the standard output of a command and inserts it into the surrounding command or variable assignment.

- Syntax:
	- POSIX/Bourne style: `$(command)`
	- Legacy: `` `command` `` (avoid when nesting)
- Prefer `$(...)` because it is easier to nest and read.

Examples:
```bash
now=$(date '+%F %T')
count=$(ls -1 -- *.txt 2>/dev/null | wc -l)
greeting=$(printf 'Hello, %s!' "$USER")
nested="$(echo "today: $(date +%A)")"
```

Quoting and word splitting:
- Always quote substitution results when assigning to a scalar to prevent word splitting and globbing: `out="$(cmd)"`.
- If you need to preserve exact lines, use an array or `mapfile/readarray` instead of splitting on IFS:
```bash
mapfile -t lines < <(printf '%s\n' "$(some-command)")
# or
readarray -t lines < <(some-command)
```

Common pitfalls:
- Using `for item in $(cmd)` splits on IFS and loses blanks; prefer:
```bash
while IFS= read -r line; do
	printf '%s\n' "$line"
done < <(some-command)
```

Performance note:
- Command substitution runs the command and captures its complete output before the shell continues, which can consume memory for very large outputs. Use process substitution or streaming loops when appropriate.

## Complex Process Replacement

Process substitution provides file-like access to the output or input of a command without creating temporary files explicitly.

- Syntax:
	- Read from command as a filename: `<(command)` — expands to a pathname (e.g., `/dev/fd/63`) that supplies `command`'s stdout.
	- Write to command as a filename: `>(command)` — expands to a pathname that the receiving program can open and write into; its input becomes `command`'s stdin.

Requirements:
- Not POSIX. Supported in Bash, Zsh, Ksh and some other shells. Under the hood it uses named pipes or `/dev/fd/*`.

Practical examples:
```bash
# Compare sorted contents of two files
diff <(sort file1.txt) <(sort file2.txt)

# Combine two streams column-wise
paste <(cut -f1 fileA) <(cut -f2 fileB)

# Use with awk to treat two inputs as files
awk 'FNR==NR{a[$1]=$0; next} $1 in a{print a[$1], $0}' <(cut -f1 file1) <(cut -f1 file2)

# Read command output without a subshell, safe for variable assignment and read loops
while IFS= read -r line; do
	process "$line"
done < <(some_long_running_command)

# Parallel consumers using tee and process substitution
some_producer | tee >(gzip -c > output.gz) >(md5sum > output.md5) > /dev/null
```

Using `>(...)` to provide writable endpoints:
```bash
# Send output simultaneously to remote and local processors
command > >(ssh user@host 'cat > /path/to/remote') 2>/dev/null
```

Behavioral notes:
- Commands receiving `<(...)`/`>(...)` see a pathname. If the consumer requires a real seekable file, process substitution may fail (named pipes are not seekable).
- The created `/dev/fd/*` or named pipe exists only for the duration of the operation; do not assume permanence.
- Use `strace` or `lsof` to inspect the actual underlying descriptor if debugging.

### Complex Process Replacement Configurations

Patterns and tips for practical configurations:

- Use process substitution to avoid temporary files when tools require filenames:
```bash
# Tools that expect filenames can work on streaming data
command_that_needs_filename "$(mktemp)"   # less elegant
# Better:
command_that_needs_filename <(producer-generating-data)
```

- Combine with `ssh` to compare remote states:
```bash
diff <(ssh host1 'ls -1 /some/dir | sort') <(ssh host2 'ls -1 /some/dir | sort')
```

- Multiple feeds to a single consumer:
```bash
awk '...' <(producer1) <(producer2) <(producer3)
```

- Use with `readarray`/`mapfile` to safely populate arrays:
```bash
mapfile -t lines < <(some-command --multiline-output)
```

- When you need a writable stream endpoint (e.g., feed a long-running process that expects a filename to write into), `>(...)` can be used:
```bash
# Make a long-running process write into a processor
long_writer_program > >(gzip -c > data.gz)
```

- Mixing stdout/stderr channels:
```bash
# Send stderr to a processor while keeping stdout to console
some_command 2> >(sed 's/ERROR:/ERR:/' >&2)
```

Caveats:
- Some commands inspect filenames or use features incompatible with pipes (e.g., expecting regular files). Test those cases; fallback to temporary files when required.
- Process substitution may behave differently across platforms (e.g., some systems lack `/dev/fd` support). Test in target environments.

### mkfifo fallback for systems without process substitution
When process substitution is unavailable, named pipes (FIFOs) provide a portable fallback. Create temporary FIFOs, run producers in the background, then consume the FIFOs as files. Use a temporary directory to avoid race conditions and ensure cleanup.

Example — portable `diff` fallback:
```bash
tmpdir=$(mktemp -d)
fifo1="$tmpdir/f1"
fifo2="$tmpdir/f2"
mkfifo "$fifo1" "$fifo2"

# Start producers in background; they write into the FIFOs
(sort file1.txt > "$fifo1") &
(sort file2.txt > "$fifo2") &

# Consumer reads from the FIFOs as if they were regular files
diff "$fifo1" "$fifo2"

# Cleanup
rm -r -- "$tmpdir"
```

Notes:
- Backgrounding the producers is required so the consumer can open the FIFOs. If a producer fails to start, the consumer will block.
- Use `mktemp -d` to create a safe temporary directory. Avoid `mktemp -u` because of race conditions.
- Ensure proper permissions on the temporary directory when running in multi-user environments.


## Here-strings

Here-strings (`<<<`) feed a single string (plus a trailing newline) to a command's stdin. They are concise for short, single-input cases.

- Syntax:
```bash
command <<< "a single line of input"
```

Common uses:
```bash
# Quick grep against a variable
grep -F 'pattern' <<< "$variable"

# Read a variable without a pipeline (no subshell)
read -r line <<< "$some_string"
printf '%s\n' "$some_string" | sed 's/foo/bar/'

# Use with commands expecting stdin
wc -w <<< "$text"
```

Differences vs heredoc:
- `<<<` is for a single string; heredocs (`<<EOF ... EOF`) are better for multi-line literal blocks.
- Here-strings append a newline automatically; heredocs preserve provided newlines exactly.

Portability:
- Here-strings are supported in Bash, Zsh, and Ksh, but are not part of POSIX sh. For maximum portability use echo/printf or heredocs.

Examples combining constructs:
```bash
# Read function output into a variable safely
read -r firstline <<< "$(some-command)"

# Process multiple lines using a here-string (small data)
while IFS= read -r line; do
	printf '%s\n' "$line"
done <<< "$small_multiline_variable"
```
