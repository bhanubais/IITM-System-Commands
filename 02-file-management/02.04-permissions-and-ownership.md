# 02.04 Permissions and Ownership

Control execution, reading, and writing authority across all filesystem nodes.

## Permission Hierarchy
Linux defines access authorities across three isolated layers.
1. **User (Owner)**: The account that engineered or took ownership of the node.
2. **Group**: The specific user group authorized to engage the node.
3. **Others**: Global access constraints applying to any external account.

## Access Matrices
Each layer possesses three toggles.
* **Read (`r`)**: Authorizes viewing file data or listing directory contents.
* **Write (`w`)**: Authorizes modifying file data or engineering and deleting nodes within a directory.
* **Execute (`x`)**: Authorizes executing a file as a program or traversing into a directory node (`cd`).

## Permission Auditing
Execute `ls -l` to audit the 10-character permission array.

* Position 1 corresponds to node type (`-` for regular/binary file, `d` for directory, `l` for soft link, `c` for character device, `b` for block device, `s` for socket, `p` for pipe).
* Positions 2 through 4 represent owner permissions.
* Positions 5 through 7 represent group permissions.
* Positions 8 through 10 represent global permissions.

Example: `-rwxr-x---` defines a standard file. The owner possesses read, write, and execute authority. The group possesses read and execute authority. Global access is denied.

## State Modifications
* `chmod` : Modify the active permission matrix of a node. Example: `chmod 755 server.py` deploys executable rights for script deployment.
* `chown` : Transfer strict ownership of a file or directory to a different user. Example: `chown nginx:wheel core.log` shifts ownership to the `nginx` user and `wheel` group.
* `chgrp` : Shift the group association acting on the target hardware block. Example: `chgrp developers shared_config.conf`.

Note: Superuser execution privileges (`sudo`) are typically required for changing ownership boundaries.
