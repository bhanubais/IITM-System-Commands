# 02.04 Link Architecture

## Purpose

Create alternate names for filesystem targets.

Understand when a link points to an inode and when it points to a path.

## Strict Syntax

```bash
ln <target_file> <hardlink>
ln -s <target_path> <symlink>
ln -s --force <new_target> <symlink>
```

## Command Map

| Command | Use |
|---|---|
| `ln` | Create a hard link. |
| `ln -s` | Create a symbolic link. |
| `ln -i` | Prompt before overwriting. |
| `ln -b` | Back up existing destination. |
| `ls -l` | Inspect symbolic link target. |
| `stat` | Inspect inode and link count. |

## Link Types

| Type | Target | Survives Target Rename | Crosses Filesystems | Can Link Directory |
|---|---|---|---|---|
| Hard link | Inode | Yes | No | Normally no |
| Symbolic link | Path text | No | Yes | Yes |

## Atomic Examples

```bash
ln data.txt data.hard
ln -s /etc/nginx/sites-available/app.conf app.conf
ln -s --force /new/target current
ln -i source.txt linked.txt
ln -b source.txt linked.txt
ls -l app.conf
stat data.txt data.hard
```

## Compound Scenarios

Enable a site through a symbolic link:

```bash
ln -s /etc/nginx/sites-available/app.conf /etc/nginx/sites-enabled/app.conf
ls -l /etc/nginx/sites-enabled/app.conf
```

Verify hard links share an inode:

```bash
ln report.txt report-copy
stat report.txt report-copy
```

Retarget a symbolic link:

```bash
ln -s --force releases/app-v2 current
ls -l current
```

## Exam Traps

Hard links are equal names for the same inode.

Symbolic links store a path.

Displayed symlink permissions are not the enforcement point. The target permissions matter.

Hard links cannot normally cross filesystem boundaries.

## Production Notes

Use symbolic links for release switching and service enablement.

Use hard links when inode identity and storage reuse are required.

Validate link targets with `ls -l` before removing or retargeting.
