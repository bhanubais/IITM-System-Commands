# 02.03 Permissions and Ownership

Control execution, reading, and writing authority across all filesystem nodes.

## Permission Hierarchy
Linux defines access authorities across three isolated layers.
1. **User (Owner)**: The account that engineered or took ownership of the node.
2. **Group**: The specific user group authorized to engage the node.
3. **Others**: Global access constraints applying to any external account.

## Access Matrices
Each layer possesses three toggles.
* **Read (`r`)**: Authorizes viewing file data or listing directory contents.
* **Write (`w`)**: Authorizes modifying file data or engineering and deleting nodes within a directory.
* **Execute (`x`)**: Authorizes executing a file as a program or traversing into a directory node (`cd`).

## Permission Auditing
Execute `ls -l` to audit the 10-character permission array.

* Position 1 corresponds to node type (`-` for regular/binary file, `d` for directory, `l` for soft link, `c` for character device, `b` for block device, `s` for socket, `p` for pipe).
* Positions 2 through 4 represent owner permissions.
* Positions 5 through 7 represent group permissions.
* Positions 8 through 10 represent global permissions.

Example: `-rwxr-x---` defines a standard file. The owner possesses read, write, and execute authority. The group possesses read and execute authority. Global access is denied.

## State Modifications

Modifying governance states requires explicit mapping of target strings. Changes can be applied singularly, sequentially, or hierarchically. Only the owner of a file (or the root user) can alter its permissions. Altering ownership typically requires superuser execution privileges (`sudo`).

### Symbolic and Structural Execution (`chmod`)
Modify the active permission matrix by binding targets `u` (user), `g` (group), `o` (others), or `a` (all) to authorities `r`, `w`, or `x`.
* `chmod u+x build.sh` : Append explicit executable rights directly to the file owner.
* `chmod o=g file.txt` : Strictly mirror the group permissions to the others block.
* `chmod g+w,o+w config.yml` : Execute a comma-separated array explicitly granting write parameters to both domains simultaneously.
* `chmod --recursive a+rX app_folder/` : Recursively grant universal read access across an entire directory tree. The capital `X` parameter intelligently grants execute permissions locally only if the target is already an executable object or a directory (which requires `x` for path traversal).

### Numeric Octal Overrides
Octal logic overrides the active permission matrix entirely using binary bitmask sequences (`4`=Read, `2`=Write, `1`=Execute).
* `chmod 755 public_html/` : Deploy `rwx` for owner (7), `r-x` for group (5), and `r-x` for others (5). This is standard operating practice for web-accessible directory structures.
* `chmod 600 server.key` : Restrict authority absolutely to the owner (`rw-`), stripping all group and global access (0, 0). Standard for SSH identity modules.
* `chmod 644 *.html` : Apply read and write exclusively to the user, and read-only to global entities, targeting all matching items in the current directory.

### Ownership Transfers
* `chown` : Transfer strict ownership of a file or directory.
  - Example: `chown nginx core.log` assigns user ownership to `nginx`.
  - Example: `chown nginx:wheel core.log` shifts ownership to the `nginx` user and explicitly binds the `wheel` group in a single pass.
  - Example: `chown -R dev_user:dev_group /var/www/html` recursively forces ownership down the entire path hierarchy to a specific account block.
* `chgrp` : Shift the group association acting on the target hardware block independently of the user layer.
  - Example: `chgrp developers shared_config.conf`.

Quick example: make a script executable and run it:
```bash
chmod 755 s1.sh   # give owner full rights and read/execute to group/others
./s1.sh           # run the script from the current directory
```
