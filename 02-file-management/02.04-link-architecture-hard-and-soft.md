# 02.04 Link Architecture (Hard and Soft)

Disks allocate stored file data using a unique structural identifier called an `inode`. A standard file path operates merely as a pointer to its underlying hardware `inode`.

You can engineer multiple access points traversing to a single physical data chunk. Pointing directly to the data's hardware `inode` constructs a **hard link**. Pointing textually to another descriptive path constructs a **soft link** (symbolic link).

## Practical Behaviors & State Inheritance

Governance states behave differently across link geometries:
- **Soft Links**: A soft link generates its own distinct configuration node. Executing `ls -l` on a symlink typically displays permissions as `lrwxrwxrwx` (777). These display metrics are completely useless. The operating system bypasses symlink permissions entirely and actively enforces the security state configured on the exact target file.
- **Hard Links**: A hard link is not a pointer; it is a cloned entrance to the exact same `inode`. Therefore, altering permissions using `chmod` against a hard link modifies the permissions of the base `inode`, universally affecting every other hard link connected to that data block simultaneously.

## Soft Links (Symbolic Links)
A soft link operates distinctly as a textual pointer routing to another absolute or relative path.
* **Failure State**: If the target file is relocated, renamed, or deleted, the soft link fractures and fails (returning a broken path exception).
* **Real World Scenario**: Configuring web application servers. Administrators generate a codebase in `/etc/nginx/sites-available/app.conf` and engineer a soft link within `/etc/nginx/sites-enabled/app.conf`. Disabling the application simply requires terminating the link without destroying the underlying repository codebase.

### Soft Link Operations
* `ln -s /path/to/target/file path/to/symlink` : Construct a soft link pointing to the defined file or directory.
* `ln -s --force /path/to/new_target path/to/symlink` : Overwrite an active symbolic link to purposefully sever its old target and redirect it.

## Hard Links
A hard link operates as an identical, equal-status anchor to the fundamental physical `inode`.
* **Failure State Shield:** Moving or renaming the originally spawned file does not fracture the link. The data natively persists securely on disk until absolutely all hard links pointing to that specific `inode` are purged via `rm`.
* **Real World Scenario:** Designing highly efficient localized backup grids. An administrator can construct daily backup folders linking completely to the exact previous day's files. The files consume zero additional hard drive space until modifications force a new specific `inode` branch to spawn.
* **Limitations:** Because they bind specifically to hardware indexes, hard links are strictly restricted to operating exclusively within the boundary of a single physical device mapping or network cluster. They cannot bridge multiple hard drives. Furthermore, standard execution actively prohibits hard-linking directories to avoid recursive loop hazards.

### Hard Link Operations
* `ln /path/to/target_file path/to/hardlink` : Construct a hard link bypassing paths and pointing straight to the target file's `inode`.

## Supplementary Link Parameters
* `ln -i` : Interactive mode. Halt and request manual authorization before overwriting active target linkage.
* `ln -b` : Generate automated backup copies of local conflict targets prior to executing the link association (designated natively with a `~` suffix).
* `ln -d` : Override structural limitation blocks to explicitly force hard-linking against directories. This must be processed natively by the superuser via `sudo`.
