# 03.02 Checksum Validation and Data Integrity

## Purpose

Verify that files arrived unchanged.

Use checksums before trusting downloaded images, binaries, archives, or local transfer payloads.

## Strict Syntax

```bash
sha256sum <file>
sha256sum -c <checksum_file>
md5sum <file>
sha1sum <file>
sha512sum <file>
```

## Command Map

| Command | Use |
|---|---|
| `md5sum` | Fast legacy hash. Not enough for strong security decisions. |
| `sha1sum` | Legacy SHA-1 hash. Avoid for new security validation. |
| `sha256sum` | Standard modern integrity check. |
| `sha256sum -c` | Verify file against checksum manifest. |
| `sha512sum` | Larger SHA-2 digest. Useful for strict integrity workflows. |

## Atomic Examples

```bash
md5sum payload.tar.gz                # legacy hash, not strong validation
sha1sum payload.tar.gz               # legacy SHA-1 digest
sha256sum payload.tar.gz             # preferred general integrity digest
sha512sum payload.tar.gz             # larger SHA-2 digest
sha256sum -c payload.tar.gz.sha256   # verify file against manifest
```

## Compound Scenarios

Validate a downloaded ISO:

```bash
curl -O https://example.com/system-agent.iso
curl -O https://example.com/system-agent.iso.sha256
sha256sum -c system-agent.iso.sha256 # expects filename and hash in manifest
```

Create a checksum manifest:

```bash
sha256sum *.tar.gz > SHA256SUMS
sha256sum -c SHA256SUMS
```

Reject mismatched payloads:

```bash
sha256sum -c payload.sha256 || { rm -f payload.bin; exit 1; }
```

## Exam Traps

Changing one byte changes the digest.

Checksum verification proves integrity against the checksum source, not trust in the checksum source itself.

MD5 is fast but unsuitable for strong security validation.

## Production Notes

Prefer SHA-256 for downloaded release artifacts.

Fetch checksums through the vendor's official channel.

Delete or quarantine payloads that fail validation.
